# Klaus — Sandboxed Claude Code

You are running inside a Klaus sandbox — a Docker container with `--dangerously-skip-permissions`.
You have `sudo` access inside this container. The project directory is mounted read-write.

> **Do not edit this file.** It is overwritten on `klaus update`. Put custom rules in `~/.claude/rules/` instead.

## Missing tools

If you need a tool that isn't installed, install it now to unblock yourself:

```bash
# Debian packages
sudo apt-get update && sudo apt-get install -y <package>

# Rust
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source ~/.cargo/env

# Python packages
pip3 install --break-system-packages <package>

# Node packages
npm install -g <package>
```

After installing, also add it to the `Klausfile` in the project root so it's pre-installed next time.

## Klausfile

The `Klausfile` lives in the project root. It uses Dockerfile syntax to pre-install tools into the container image. Just write `RUN`, `ENV`, etc. instructions.

Everything runs as the `klaus` user (which has passwordless sudo), so use `sudo` for system-level commands — same as in a live session.

Example:

```dockerfile
RUN sudo apt-get update && sudo apt-get install -y postgresql-client
RUN curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
ENV PATH="/home/klaus/.cargo/bin:${PATH}"
RUN pip3 install --break-system-packages numpy pandas cvxpy
```

When you edit the Klausfile, the changes take effect on the next `klaus` invocation (the container is rebuilt). For the current session, install tools directly as described above.

## Networking

You are running inside a Docker container. `localhost` here refers to the container, not the host machine. If you can't reach a service the user says is running on their machine, walk them through these steps:

1. **Use `host.docker.internal` instead of `localhost`** — this hostname resolves to the host machine. Try `curl http://host.docker.internal:<port>` to test.

2. **If that also fails, the server probably binds to `127.0.0.1` only.** Most dev servers do this by default. Tell the user they need to restart the server bound to `0.0.0.0` (e.g. `--host 0.0.0.0` for Vite/Next, `--bind 0.0.0.0` for Python).

3. **Once the server is reachable via `host.docker.internal`**, you can make `localhost` work too using socat (useful for tools like playwright-cli that hardcode `localhost`):
   ```bash
   # socat proxy — works with both IPv4 and IPv6 host addresses
   socat TCP-LISTEN:<port>,fork,reuseaddr TCP:host.docker.internal:<port> &
   ```
   If socat isn't installed: `sudo apt-get update && sudo apt-get install -y socat`

## Browsers

Headless Chrome/Chromium must be launched with `--no-sandbox` inside this container. The Docker container itself is the sandbox — Chrome's internal sandbox conflicts with the container's security restrictions. When using Playwright, Puppeteer, or launching Chrome directly, always pass `--no-sandbox`:

```bash
# Playwright
npx playwright install chromium
PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH=$(npx playwright install chromium 2>/dev/null && echo "installed") \
  npx playwright test --launch-options='{"args":["--no-sandbox"]}'

# Direct Chrome
chromium --no-sandbox --headless
```

## Limitations of the sandbox

You cannot modify your own sandbox. The `klaus` script, Dockerfile, base image, and this rules file all live outside the container. If you encounter a limitation that requires changes to how the container is configured (e.g. additional docker run flags, base image changes, entrypoint modifications), explain the issue to the user and suggest they:

1. Read the Klaus source at `~/.klaus/install/` (it's a full git clone) to understand the current setup — look at `Dockerfile`, `klaus` (the main script), and this file
2. Write up what needs to change and why, with specific file edits if possible
3. Tell the user to either:
   - Edit the files in `~/.klaus/install/` on the host and submit a PR from there (`cd ~/.klaus/install && git checkout -b fix-name && ... && gh pr create`)
   - Or open an issue at https://github.com/jtremback/klaus with your writeup

Things you *can* change from inside the container:
- Install tools at runtime with `sudo` (and add them to the Klausfile for next time)
- Edit project files (the mounted working directory)
- Add custom rules in `~/.claude/rules/` (persists across sessions)

Things you *cannot* change:
- Docker run flags (networking, capabilities, volume mounts)
- The base image or Dockerfile
- This rules file (overwritten on `klaus update`)
